# 需求文档研讨小作坊 · PRD Discussion Workshop

一个用“需求研讨会”的方式生成 PRD 的小工具。

它不是让 AI 上来直接甩你一份长文档，而是让几个 AI 角色在一个聊天框里一起讨论需求：产品负责收束，用户负责挑体验问题，研发负责看实现风险，领导负责盯目标和边界。你作为真正的需求负责人，随时拍板、纠偏、补充信息。左侧会同步沉淀成 PRD。

简单说：右边把需求聊清楚，左边把文档写出来。

## 它解决什么问题

很多 PRD 的问题不是“不会写”，而是前面没聊清楚：

- 背景太泛，不知道到底要解决什么。
- 流程没收束，研发看完还是不知道怎么做。
- 文案只写“建议展示”，最后还要重新确认。
- 异常、状态、保存、生效规则经常遗漏。
- AI 一次性生成时容易脑补，写得很完整但不一定真实。

这个项目的思路是：先用对话把问题问出来，再把确认过的内容写进 PRD。一步步用AI收束AI。

## 核心优势

### 多角色不是群聊刷屏

内置 4 个角色：

- 资深产品：把模糊想法收束成可落地流程。
- 用户：从真实使用场景里找不顺手的地方。
- 研发：提前指出状态、接口、异常和实现成本。
- 领导：关注目标、优先级和首版范围。

它们不是每次你说一句就四个人一起回复。系统会根据上一句内容判断现在更应该谁发言，尽量模拟真实需求讨论里的接力感：有人补漏洞，有人追问，有人收束，有人提醒别做太大。

### PRD 会跟着讨论实时变化

左侧 PRD 不是静态模板，而是根据讨论内容不断更新。

默认规则比较克制：

- 多个需求会拆成多个一级标题。
- 单个需求不会堆一堆没用章节。
- 背景、目标、使用场景合并写，避免套话。
- 需求主体按用户操作顺序写清楚入口、流程、反馈、文案、异常、保存和生效。
- 待确认问题不会写进 PRD，而是在聊天里继续问你。
- 如果涉及文案选择，会先在聊天里给你几个方案和理由，等你拍板后再进入 PRD。

这样生成出来的 PRD 更像“可以交付给研发看的需求”，而不是“看起来很完整的 AI 作文”。

### Prompt 可以改，项目背景可以换

角色 Prompt 和 PRD Prompt 都能在页面里改。

所以这个工具不应该只绑定某一个产品线。你可以把角色背景换成别的团队，把 PRD 格式换成你们自己的规范。底层逻辑负责调用这些 Prompt，而不是把某个项目的规则写死。

### Skill 放后台，少重复废话

项目把一些稳定能力放到了 `backend-skills` 里，比如：

- 多角色对话怎么接力。
- PRD 应该怎么组织。
- Wiki 知识库怎么检索。
- 怎么把 PRD 转成原型图提示词。
- 怎么提取产品内文案做多语言翻译。

这么做的好处是，不用每轮对话都把一大段规则重新塞给模型，后续也方便按能力单独调整。

## 搭配 Wiki 知识库会更好

这个项目可以单独用，但更推荐搭配团队自己的 LLM Wiki 知识库。

为什么？因为很多需求不是凭空来的，它会依赖：

- 以前的 PRD。
- 产品术语。
- 历史版本规则。
- 接口和字段定义。
- 品牌、市场、运营口径。
- 会议纪要和历史决策。

如果没有 Wiki，AI 只能根据当前对话推理。如果接入 Wiki，它可以先查团队已有知识，再参与讨论，幻觉会少很多，也不需要你反复解释历史背景。

推荐的 Wiki 形态是：

- 有一个 `index.md` 做全局目录。
- Wiki 页面是大模型整理后的知识精华，不是原始资料堆叠。
- 查询时先看 index，再精确读取相关页面或 chunk。
- 检索结果作为只读证据包进入对话，AI 不能编造 Wiki 里没有的内容。

当前项目支持这些 Wiki 接口：

- `GET /api/wiki/manifest`
- `GET /api/wiki/search?q=...&limit=...`
- `GET /api/wiki/chunk/:chunkId`
- `GET /api/wiki/doc/:docId`

## 现在已有的功能

- 多角色需求讨论。
- 左侧实时 PRD。
- 修改角色 Prompt。
- 修改 PRD Prompt。
- 生成原型图 Prompt。
- 提取产品内文案并生成多语言翻译交付表。
- 保存当前需求讨论快照。
- 恢复历史需求讨论。
- 接入局域网 Wiki 检索。
- 简单记录使用情况。
- 支持局域网分享给同事使用。

## 怎么运行

需要 Node.js 18 或更高版本。

```bash
npm start
```

默认地址：

```text
http://localhost:4188
```

如果要给局域网里的同事用，可以用你电脑的局域网 IP 访问，例如：

```text
http://10.10.116.72:4188
```

## 配置模型 API

复制 `.env.example` 为 `.env`，然后填你的中转 API。

```env
CODEX_API_KEY=your-relay-key
CODEX_API_BASE_URL=https://your-codex-api-relay.example.com/v1
CODEX_API_MODEL=gpt-5.5
CODEX_API_MODE=responses
```

如果你的中转站走 `/v1/chat/completions`：

```env
CODEX_API_MODE=chat_completions
CODEX_API_PATH=/chat/completions
```

如果你的中转站走 `/v1/responses`：

```env
CODEX_API_MODE=responses
CODEX_API_PATH=/responses
```

项目优先读取 `CODEX_API_*`。`OPENAI_*` 只是兼容兜底，不要求直连 OpenAI。

## 配置 Wiki

如果要启用局域网 LLM Wiki：

```env
WIKI_ENABLED=true
WIKI_BASE_URL=http://10.10.124.11:8787
WIKI_SEARCH_PATH=/api/wiki/search
WIKI_DOC_PATH=/api/wiki/doc
WIKI_CHUNK_PATH=/api/wiki/chunk
WIKI_INDEX_DOC_ID=wiki/index.md
WIKI_TOKEN=
```

如果 Wiki 服务启用了 token：

```env
WIKI_TOKEN=your-wiki-token
```

## 保存需求

讨论到一半可以保存。

保存后会按需求标题创建一个文件夹，里面会有：

- `.session.json`：完整会话状态。
- `.prd.md`：当前 PRD。
- `.discussion.md`：讨论记录。

下次输入 `.session.json` 路径就能继续聊。

后面调试稳定后，会继续加原型图 MCP 接入。

现在已经能把 PRD 转成给图像模型看的原型图 Prompt。下一步希望直接接入原型图 MCP，让它可以根据确认后的 PRD 自动生成可评审的界面原型。

理想状态是：

需求讨论 -> PRD 沉淀 -> 原型图生成 -> 文案翻译

这几步都在一个工作台里完成。

## 总结

这个项目更像一个“AI 需求评审搭子”，而不是单纯的 PRD 生成器。

它的重点是：

- 先讨论，再写文档。
- 先确认，再沉淀。
- 用角色互相挑问题，减少一个模型自说自话。
- 用 Wiki 接住团队历史知识，减少重复解释和幻觉。
- 用可编辑 Prompt 保持迁移性。

如果你经常遇到“需求没聊清楚，但 PRD 已经写得很长”的情况，这个工具就是为这种场景准备的。
